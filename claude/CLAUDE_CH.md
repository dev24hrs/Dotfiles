## 核心原则：推理先行 (Reasoning-First)

**在开始编写、修改或重构任何代码之前，必须启动多步推理流（Long-horizon loop）。禁止直接输出代码。**

### 适用条件

三阶段推理对**任何代码修改都是强制性的**，无论规模或复杂程度。包括但不限于：

- 新功能实现、涉及多文件的修改、架构或设计变更
- 性能优化、重构、安全相关改动
- 单行修复、拼写纠正、配置变更、revert/rollback 操作

以下场景可以跳过（未修改代码）：纯信息查询、代码阅读/解释、不改变代码行为的拼写错误。

### 阶段一：Self-Reflection（全面反思与上下文审查）

在动笔之前，对需求、代码库现状及历史上下文进行深度解构：

- **需求对齐**：用你自己的话重新阐述用户核心诉求，确保无理解偏差
- **上下文审查**：检查受影响文件、依赖关系、全局状态及现有架构设计
  - 对将被新增/修改/删除的符号（函数、类型、变量、配置项等），必须先用搜索工具（`rg`/`grep` 等）定位其**全部**引用/调用/传参位置，列出文件路径（必要时附行号），禁止凭印象判断"应该没人用到"
  - 签名变更（参数列表、返回值类型等）必须确认所有调用方是否需同步修改，并在分析中列出受影响的调用点
  - 删除符号前必须用搜索结果证明引用数为 0（测试代码、脚本、文档示例均需计入），而非主观判断
  - 反例（禁止）：「这个函数看起来没人用，直接删了」（未经搜索验证）
  - 正例（要求）：「搜索发现 `parseConfig` 在 3 处被调用：`src/loader.rs:42`（直接调用）、`tests/config_test.rs:18`（测试用例）、`src/cli/mod.rs:7`（trait 间接调用），修改签名后已同步更新这 3 处并补充测试」

- **纠错与破局**：之前的实现是否有盲点？用户提议是否有逻辑漏洞或破坏既有设计模式？寻找隐藏的暗坑并提出修正方案

### 阶段二：Side-effects（副作用评估）

评估本次代码变更对整个系统的影响，包括但不限于：

- **性能**：时间/空间复杂度上升、内存泄漏、I/O 阻塞、高并发下的竞争条件
- **破坏性变更**：API 契约、数据库 Schema、公用组件、测试用例是否失效
- **边界情况**：空值、网络异常、极值、并发请求、国际化等场景
- **安全与维护性**：是否引入安全漏洞、增加技术债或降低可读性

### 阶段三：Precision Implementation（高精度代码实现）

前两个阶段推演完成并得出明确结论后，方可进入代码编写：

- **严谨性**：代码必须完整，禁止 `// TODO` 或 `// 省略其余代码...` 等伪代码占位符
- **防御性编程**：加入必要的错误处理（Error Handling）和边界防御
- **可测试性**：确保代码逻辑清晰，易于编写单元测试

### 输出格式

回答结构应与以上三阶段保持一致：先输出反思与副作用分析，最后给出代码。如需跳过推理流程，简要说明原因即可。

## 现代化与标准库优先原则

在生成任何语言的代码时，遵循以下优先级：

1. **标准库优先**：当标准库能提供足够解决方案时，优先使用标准库而非第三方依赖。引入新依赖需要说明理由（标准库方案缺失、存在缺陷、或过于冗长）。
2. **现代惯用法优于旧式写法**：使用语言现代/推荐的惯用法（2020 年后的写法），而非早期版本的旧式模式。包括但不限于：
   - 函数声明（箭头函数、简洁方法语法等）
   - 集合操作（当可读性足够时，优先使用声明式/函数式 API 而非手动循环）
   - HTTP / I/O 操作（优先使用现代客户端 API 而非已废弃的旧 API）
   - 错误处理（优先使用语言原生模式而非手动拼凑的变通方案）
   - 类型系统（优先使用新式类型构造，而非 `any`/`Object`/`void*` 等逃逸手段）
3. **版本感知**：以项目工具链/配置文件（如 `go.mod`、`pyproject.toml`、`.nvmrc`）中声明的语言版本为准。不使用高于项目目标版本的新特性，也不使用项目目标版本中已废弃的写法。
4. **语言细节参见专项文档**：具体语言的对照表参见 `@CLAUDE_<Lang>.md`。

## 强制验证规则

以下规则无条件适用。不得因感知到的简单性或熟悉度而例外。

### 规则 1：Revert / Rollback 前必须先读取

在执行任何涉及 **revert、rollback、undo、restore 或"恢复为原始版本"** 的操作之前，必须先用 Read 确认目标文件的当前状态。永远不要依赖对之前回合中文件内容的记忆或印象。

- 反例（禁止）："我记得原始版本是 X，直接写回 X。"（未经验证）
- 正例（要求）：Read 文件 → 确认当前状态 → 然后写入正确的原始版本。

### 规则 2：编辑后验证

在每次 Edit 或 Write 操作之后，立即用针对性的 Read 或 grep 验证变更行。在继续下一步之前，确认最终状态符合预期。

### 规则 3：代码修改前必须定位根因

在调试行为异常（错误状态、意外输出、静默失败）时，必须先用诊断命令捕获真实运行时数据（例如检查实时输出、查询日志、检查实际变量值）来定位根因。在识别并展示导致异常的实际数据之前，不要修改检测逻辑、正则表达式或条件分支。

- 反例（禁止）："也许正则太宽泛了，让我缩小范围。" → 无数据就编辑。
- 正例（要求）：捕获被匹配或比较的实际值 → 找到导致不匹配的确切数据 → 然后根据证据修复。
